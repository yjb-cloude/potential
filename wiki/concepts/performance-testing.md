# 性能测试

## 核心术语

- **QPS (TPS)**：每秒查询数 / 每秒事务数，衡量系统吞吐能力的核心指标
- **响应时间（RT）**：从发送请求到收到完整响应的耗时，通常关注平均值和百分位值
- **吞吐量**：单位时间内系统处理的请求量，与 QPS 相近但范围更广
- **并发用户数**：系统同时处理的活跃用户数，不等于连接数
- **P99 / P95**：99% / 95% 的请求响应时间低于该值，反映长尾延迟

## 测试类型

1. **基准测试（Baseline）**：低并发下确定正常响应时间基线
2. **负载测试（Load Test）**：模拟预期并发，验证系统是否满足性能目标
3. **压力测试（Stress Test）**：逐步增加负载找出系统崩溃点（瓶颈点）
4. **稳定性测试（Endurance / Soak Test）**：长时间运行（4-24小时），检测内存泄漏和资源耗尽

## JMeter 核心组件

- **线程组（Thread Group）**：设置并发用户数、Ramp-Up 时间、循环次数
- **Sampler**：HTTP 请求采样器，配置协议、域名、路径、方法、参数
- **断言（Response Assertion）**：验证响应状态码/文本内容，标记失败请求
- **监听器（Listener）**：
  - **聚合报告（Summary Report / Aggregate Report）**：平均/中位/90%/95%/99% 响应时间、吞吐量、错误率
  - **结果树（View Results Tree）**：调试时查看每个请求的完整请求/响应
  - **TPS 曲线图（jp@gc - Transactions per Second）**：实时 TPS 变化趋势
- **配置元件**：
  - **CSV Data Set Config**：从 CSV 文件读取参数化数据
  - **HTTP Request Defaults**：设置公共请求默认值（域名、Header）
  - **User Defined Variables**：全局自定义变量

## 参数化

使用 **CSV Data Set Config** 实现大规模用户数据参数化，支持多列读取、随机顺序、线程独立副本。配合 **ForEach Controller** 实现循环数据消费。

## 性能瓶颈定位路径

典型的排查顺序：

```
硬件资源（CPU / 内存 / 磁盘 IO / 网络带宽）
    → 数据库慢 SQL（通过慢查询日志 + EXPLAIN 分析）
    → 缺少缓存（Redis / 本地缓存命中率低）
    → 连接池不足（数据库连接池 / HTTP 连接池耗尽）
    → 代码逻辑（锁竞争 / 串行化 / 大对象分配 / 频繁 GC）
```

## 性能测试报告关键指标

| 指标 | 说明 | 关注点 |
|------|------|--------|
| 平均 RT | 所有请求平均响应时间 | 整体体验 |
| P95 / P99 | 长尾延迟 | 极端场景体验 |
| QPS / TPS | 每秒处理能力 | 系统容量 |
| 错误率 | 失败请求占比 | 稳定性 |
| CPU / 内存 | 资源占用率 | 资源瓶颈 |
| GC 频率 | JVM 垃圾回收 | 内存健康 |
