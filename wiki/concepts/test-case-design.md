# 测试用例设计

## 设计方法

### 1. 等价类划分

将输入域划分为若干个**等价类**，每个等价类中的任意值代表同等的测试效果。

- **有效等价类**：符合需求的合法输入
- **无效等价类**：不符合需求的非法输入

每条测试用例至少覆盖一个有效等价类，每条无效等价类单独一条用例（确保异常分支准确触发）。

### 2. 边界值分析

大量缺陷集中在输入域的边界附近。核心思路：取 **min / min+ / max / max-** 进行测试。

如输入范围 1-100，测试值：0、1、2、99、100、101。

### 3. 判定表

适合处理**多条件组合**场景。结构：

- **条件桩**：列出所有输入条件
- **动作桩**：列出所有可能的输出
- **条件项**：各条件的取值（Y/N）
- **动作项**：对应条件下的输出

典型应用：订单状态流转、权限判断、优惠规则引擎。

### 4. 因果图法

从**原因（输入条件）→ 结果（输出）** 的逻辑关系出发，推导出测试用例。适合逻辑关系复杂的场景，通常辅助判定表使用。

### 5. 正交实验法（Pairwise 测试）

用最少的用例覆盖最多的参数组合。工具推荐：**PICT（Microsoft）** 或 **AllPairs**。如 3 个参数各 3 个取值，全组合 27 种，Pairwise 可压缩到 9 种。

### 6. 场景法

基于**用户操作流程**设计用例：

- **基本流**：正常操作路径（Happy Path）
- **备选流**：异常或分支路径（如取消、回退、错误输入）
- 适用场景：业务流程测试、端到端测试

### 7. 错误推测法

凭借测试经验和直觉，推测系统可能出错的地方：空值、超长字符串、特殊字符、边界并发、重复提交等。不系统但高效，是其他方法的补充。

## 测试用例要素

一个标准的测试用例应包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| 用例编号 | 唯一标识 | TC-LOGIN-001 |
| 标题 | 清晰描述测试场景 | 使用正确账号密码登录 |
| 前置条件 | 执行前的必要条件 | 已注册用户，账号已激活 |
| 测试步骤 | 操作步骤（编号） | 1.打开登录页 2.输入账号 3.输入密码 4.点击登录 |
| 测试数据 | 具体输入值 | 账号: admin / 密码: 123456 |
| 预期结果 | 期望的系统行为 | 登录成功，跳转首页 |
| 优先级 | P0~P3 等级 | P0（冒烟测试必过） |

## 覆盖率评估

- **需求覆盖**：每条需求对应多少条测试用例
- **代码覆盖**：
  - **行覆盖**：每行代码是否被执行
  - **分支覆盖**：if/else 各分支是否覆盖
  - **路径覆盖**：所有可能执行路径是否覆盖
  - **条件覆盖**：布尔表达式中每个条件的 True/False
- 工具：Jacoco（Java）、Gcov（C/C++）、Istanbul（JavaScript）
