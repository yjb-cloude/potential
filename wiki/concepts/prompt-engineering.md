# Prompt Engineering

Prompt Engineering 是设计和优化输入提示（Prompt）以引导大语言模型（LLM）生成期望输出的技术。它是 AI 应用开发中最基础也最重要的能力之一。

## 三种角色

LLM 的对话结构由三种角色构成：

- **System** — 设定 AI 的行为、身份和约束。系统提示词在对话开始时设定，对后续所有交互生效。
- **User** — 用户的输入，即实际的问题或指令。
- **Assistant** — 模型的回复。Assistant 的输出也可以作为后续对话的上下文输入。

## 基础模式

- **Zero-shot Prompting** — 直接提问，不给示例。适用于模型已掌握的知识，例如"把这段文本翻译成英文"。
- **Few-shot Prompting** — 先给若干示例，再提问题。示例能帮助模型理解输出格式和预期风格。
- **Chain-of-Thought（CoT）** — 引导模型展示推理过程，常用"让我们逐步思考"或"Let's think step by step"。这种方法能显著提升复杂推理任务的准确性。

## 结构化输出

通过明确的格式约束让 LLM 输出结构化数据：

- 指定 JSON 格式输出，定义好字段和类型
- 要求输出 Markdown 表格
- 枚举值约束（限定输出只能是给定的几个值之一）

## 进阶技巧

- **角色设定（Role Prompting）**：开头即设定角色，例如"你是一名资深测试工程师"，让模型进入特定专业领域模式。
- **分步指令（Step-by-Step）**：用编号分步骤描述任务，降低遗漏风险。
- **格式约束（Format Constraint）**：明确要求格式，例如"只返回 JSON，不要解释"。
- **负面提示（Negative Prompting）**：告诉模型"不要做什么"，例如"不要输出多余的说明"，"不要编造数据"。

## 参数调节

- **temperature（0-2）**：越低输出越确定和可复现；越高输出越多样和创造性。做测试用例生成通常用 0。
- **top_p**：核采样参数，控制候选词汇的累计概率阈值。
- **max_tokens**：限制模型输出的最大 token 数。

## 实践建议

- **Prompt 版本管理**：将 prompt 视同代码，使用 Git 管理版本变更。
- **Prompt 测试**：建立测试用例集，在不同 prompt 版本上运行对比，量化效果。
