# 测试自动化体系

## 测试金字塔（Test Pyramid）

```
         /\
        /  \         UI 自动化（端到端）
       /    \        （Selenium / Playwright / Cypress）
      /------\
     /        \      接口自动化（API）
    /          \     （Rest Assured / Postman / JMeter）
   /------------\
  /              \   单元测试（Unit Test）
 /                \  （JUnit / TestNG / Mockito）
/------------------\
```

- **单元测试**：测试最小代码单元（方法/类），开发阶段执行，速度快
- **接口测试**：测试 API 接口，覆盖业务逻辑校验，性价比最高
- **UI 测试**：模拟用户操作，覆盖端到端流程，执行慢且脆弱

## 测试类型分类

| 维度 | 类型 | 说明 |
|------|------|------|
| 阶段 | 单元测试 / 集成测试 / 系统测试 / 验收测试 | 按开发阶段划分 |
| 是否运行代码 | 静态测试 / 动态测试 | 代码审查 vs 实际运行 |
| 是否关注内部结构 | 白盒 / 黑盒 / 灰盒 | 结构 vs 功能 vs 结合 |
| 自动化程度 | 手工 / 半自动 / 全自动 |  |
| 目的 | 功能 / 性能 / 安全 / 兼容 / 回归 / 冒烟 |  |

## 自动化测试适用场景

✅ **适合自动化**：
- 回归测试（每次发布都要跑）
- 冒烟测试（构建验证）
- 数据驱动测试（大量数据组合）
- 重复性操作（频繁执行的场景）
- 跨平台/跨浏览器测试

❌ **不适合自动化**：
- 一次性测试
- 探索性测试（需要人的判断力）
- UI 频繁变动的场景
- 安装/设置类测试（环境依赖多）

## 自动化框架类型

| 类型 | 示例 | 适用场景 |
|------|------|---------|
| 线性框架 | 录制回放 | 简单场景，快速原型 |
| 模块化框架 | 按功能拆分类 | 中小型项目 |
| 数据驱动框架 | 外部数据源驱动 | 需要大量数据组合 |
| 关键字驱动 | 用关键字描述操作 | 非技术人员编写用例 |
| 混合框架 | 上述组合 | 企业级项目 |
