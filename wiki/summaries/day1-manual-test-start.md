## 今天做了什么

### 🔵 手工测试——saucedemo.com

**登录功能测试：**

| 场景 | 测试数据 | 结果 | 分析 |
|------|---------|------|------|
| 正确登录 | standard_user / secret_sauce | ✅ 登录成功 | 冒烟通过 |
| 空用户名 | "" / secret_sauce | ✅ Username is required | 前端非空校验 |
| 空密码 | standard_user / "" | ✅ Password is required | 前端非空校验 |
| 锁定用户 | locked_out_user / secret_sauce | ❌ 登录失败，提示已锁定 | 系统预留的测试账号 |
| 不存在用户 | admin / admin123 | ❌ do not match any user | 安全设计：不暴露"用户不存在" |

**发现亮点**：
- 错误提示没有区分"用户不存在"和"密码错误"，这是安全设计 ✓
- 登录按钮在空字段时没有禁用，可以尝试点
- locked_out_user 是用于测试"账户锁定"场景的预留账号

## 学到的新概念
1. **等价类划分**（初步体会）：有效输入（正确账号）和无效输入（错误密码/空字段）
2. **安全测试思维**：错误提示不应该透露太多信息
3. **测试数据设计**：系统内置了不同类型的测试账号（标准用户、锁定用户、问题用户等）
