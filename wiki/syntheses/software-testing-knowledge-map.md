## 一、测试理论基础

### 软件质量模型

**ISO/IEC 25010** 是业界通用的软件质量模型，定义了以下八大质量特性：

| 质量特性 | 说明 | 典型测试关注点 |
|---------|------|--------------|
| 功能性 (Functional Suitability) | 功能正确、完整、恰当 | 功能测试、边界验证 |
| 性能效率 (Performance Efficiency) | 时间行为、资源利用率 | 负载测试、压力测试 |
| 兼容性 (Compatibility) | 共存、互操作 | 跨浏览器、跨版本测试 |
| 易用性 (Usability) | 可识别、可学习、可操作 | 用户体验测试、A11y |
| 可靠性 (Reliability) | 成熟度、可用性、容错 | 稳定性测试、故障恢复 |
| 安全性 (Security) | 机密性、完整性、抗抵赖 | 渗透测试、SQL 注入 |
| 维护性 (Maintainability) | 模块化、可复用、可分析 | 代码审查、静态分析 |
| 可移植性 (Portability) | 适应性、可部署 | 环境兼容测试 |

**面试重点**：至少能说出 5 个质量特性及其测试含义，面试时用"我们项目主要关注功能、性能和安全"来引出实际经验。

### 测试类型

- **功能测试** — 验证功能是否符合需求，是测试工作的主体
- **性能测试** — 验证响应时间、吞吐量、资源使用（详见 [[performance-testing]]）
- **安全测试** — 发现系统安全漏洞（XSS、CSRF、SQL 注入、越权）
- **兼容性测试** — 不同浏览器/OS/设备/分辨率下正常工作
- **易用性测试** — 用户操作是否直观、高效，符合使用习惯
- **可靠性测试** — 系统在长时间/高负载下是否稳定，异常恢复能力
- **回归测试** — 修改后验证原有功能不受影响，自动化最擅长的场景
- **冒烟测试** — 构建后对核心功能快速验证，决定是否进入详细测试

**面试重点**：能区分"功能 vs 性能 vs 安全"的核心差异，说明回归测试的重要性。

### 测试级别

```
需求分析 → 单元测试 → 集成测试 → 系统测试 → 验收测试
                              ↓
                          UAT (用户验收)
```

| 级别 | 测试对象 | 测试人 | 典型时间 |
|------|---------|-------|---------|
| 单元测试 (Unit) | 方法/函数/类 | 开发 | 编码阶段 |
| 集成测试 (Integration) | 模块间接口 | 开发/测试 | 模块联调 |
| 系统测试 (System) | 完整系统 | 测试 | SIT 阶段 |
| 验收测试 (UAT) | 业务流程 | 用户/产品 | 上线前 |

**面试重点**：每个级别的定位和负责方，特别是"集成测试和系统测试的区别"是高频题。

### 测试方法

| 维度 | 分类 | 说明 |
|------|------|------|
| 是否关注内部结构 | **白盒** (结构逻辑) / **黑盒** (功能) / **灰盒** (结合) | 面试常问区别 |
| 是否运行代码 | **静态** (评审/走查) / **动态** (执行) | 静态发现 30% 以上的缺陷 |
| 执行方式 | **手工** / **自动化** | 手工做探索，自动化做回归 |
| 测试策略 | 正向 (验证能做) / 逆向 (验证防错) | 异常场景覆盖 |

**面试重点**：白盒 vs 黑盒的区别和适用场景是必考题。

### 测试设计方法

| 方法 | 核心思想 | 适用场景 |
|------|---------|---------|
| **等价类划分** | 将输入域分为有效/无效等价类，每个类取一个代表值 | 所有输入框测试 |
| **边界值分析** | 取边界值（最小值、最大值、略超边界） | 数值范围输入 |
| **因果图法** | 用逻辑图表示输入条件和输出结果的关系 | 组合逻辑复杂的场景 |
| **判定表** | 条件与动作的组合矩阵，覆盖所有逻辑路径 | 业务规则较多（如审批流程）|
| **正交实验法** | 用正交表选择最少的组合覆盖最多因素 | 多参数组合（如筛选条件）|
| **场景法** | 基于业务流程的路径分析 | 端到端业务流程 |
| **错误推测法** | 凭经验推测可能出Bug的地方 | 补充用例，经验依赖强 |

**面试重点**：等价类划分 + 边界值分析是最常考的组合，面试常给一个输入条件让你现场设计用例。场景法在业务测试中应用最广。

### 测试流程

```
需求分析 → 测试计划 → 用例设计 → 用例评审 → 执行测试 → 缺陷管理 → 测试报告
    ↓          ↓           ↓                    ↓             ↓          ↓
 理解需求   评估工期   编写用例             记录结果     跟踪Bug   总结质量
 识别风险   制定策略   评审用例             执行用例     回归验证   给出结论
```

**面试重点**：能完整说出一条测试流程，并能具体描述每个阶段的产出物。

## 二、测试管理

### 测试计划要素

一份完整的测试计划应包含：

1. **测试范围** — 测什么、不测什么（In Scope / Out of Scope）
2. **测试策略** — 各类型测试的方法和工具选择
3. **资源安排** — 人员、环境、数据准备
4. **时间计划** — 各阶段的起止时间、里程碑
5. **准入/准出标准** — 开始测试和结束测试的条件
6. **风险及应对** — 识别风险，制定 mitigation 方案
7. **交付物** — 测试用例、Bug 报告、测试报告等

**面试重点**：提到"准入标准"和"准出标准"会加分，这是区分新手和有经验者的细节。

### 测试用例编写规范

标准的测试用例包含以下要素：

```
标题：简短描述测试场景（【模块】_测试点_条件）
前置条件：测试开始前必须满足的条件
测试步骤：清晰的操作步骤，编号排列
预期结果：具体可验证的结果描述
"""

示例：
标题：【登录】_密码错误_输入错误密码登录
前置：用户已注册，数据库中有该账号
步骤：1. 打开登录页面 2. 输入正确用户名 + 错误密码 3. 点击登录
预期：页面提示"用户名或密码错误"，不会跳转
```

**优秀用例的特征**：
- **原子性**：一个用例只测一个点
- **可复现**：严格按照步骤执行一定能复现
- **可判定**：预期结果明确，任何测试人员执行结果一致
- **独立性**：用例之间不互相依赖

**面试重点**：面试官可能会给你一个功能，让你现场口述测试用例设计思路。

### Bug 生命周期

```
New (新提交)
  │
  ▼
Assigned (已分配) ←─── Rejected (驳回：非Bug/重复/无法复现)
  │
  ▼
Fixed (已修复)
  │
  ▼
Verified (已验证)
  │
  ▼
Closed (已关闭)
  │
  └───→ Reopen (重新打开：验证不通过)
```

**关键的流转规则**：
- 开发 Fix 后必须指定给测试验证，不能直接 Close
- 测试验证不通过则 Reopen，回到 Assigned
- Rejected 需要有充分理由，测试有异议可以提出来讨论

**面试重点**：Bug 生命周期流程几乎必问，要能用流程图口述清楚。

### Bug 严重级别 vs 优先级

| 级别 | 严重程度 (Severity) | 优先级 (Priority) |
|------|-------------------|-------------------|
| P0/Critical | 系统崩溃/主功能不可用 | 立即修复，阻塞发布 |
| P1/Major | 功能无法正常使用 | 本版本必须修复 |
| P2/Minor | 功能可用但体验不佳 | 本版本应修复 |
| P3/Trivial | 界面问题/文字错误 | 可以延期/下一版本 |

**核心区别**：
- **Severity**：Bug 对系统的破坏程度（技术角度）
- **Priority**：Bug 被修复的紧急程度（业务角度）
- 高 Severity 不一定高 Priority（如"主页Logo显示错位" Severity 低但如果是大客户要求则 Priority 高）
- 低 Severity 不一定低 Priority（如"用户名需要加特殊字符校验"虽小但必须改）

**面试重点**：能举例说明 Severity 和 Priority 的四种组合情况是加分项。

### 测试报告模板

```
XX 项目 V1.0 功能测试报告

1. 测试概述：测试范围、时间、人员
2. 测试环境：硬件配置、软件版本、浏览器
3. 测试统计：总用例数、通过/失败/阻塞数、通过率
4. 缺陷统计：Bug 总数、按严重级别分布、按模块分布
5. 遗留问题：已知但未修复的 Bug 及影响评估
6. 测试结论：是否达到上线标准，建议是否发布
7. 风险提示：已知风险和未来建议
```

**面试重点**：提到"遗留问题分析"和"上线建议"能体现质量意识。

### 测试度量指标

| 指标 | 公式 | 含义 |
|------|------|------|
| **用例通过率** | 通过用例数 / 总执行用例数 | 反映当前版本质量 |
| **缺陷密度** | Bug 数 / 代码行数(或功能点数) | 每千行代码 Bug 数，评估模块质量 |
| **用例覆盖率** | 已覆盖需求数 / 总需求数 | 功能点覆盖率，衡量充分性 |
| **自动化覆盖率** | 自动化用例数 / 总用例数 | 标记回归效率 |
| **缺陷发现率** | 测试发现的 Bug / (测试发现 + 线上发现) | 测试质量评估 |
| **需求稳定性** | 需求变更次数 | 影响测试计划和用例设计 |

**面试重点**："用例通过率"和"缺陷密度"是最常用的两个指标。面试官会问"你们团队用什么指标衡量质量"。

## 三、自动化测试

### 测试金字塔与框架类型

详见 → [[test-automation-types]]

核心要点记忆：

```
         /\
        /  \         UI 自动化（慢、贵、维护成本高）
       /    \
      /------\
     /        \       API 自动化（性价比最高）
    /          \
   /------------\
  /              \    单元测试（快、便宜、覆盖面广）
/------------------\
```

- **单元测试**占比 60-70%，**API 测试**占比 20-30%，**UI 测试**占比 5-10%
- API 自动化是性价比最高的分层，投入产出比最优
- 框架类型：线型 → 模块化 → 数据驱动 → 关键字驱动 → 混合框架

### [[selenium-webdriver]]

- **元素定位**：ID > Name > CSS Selector > XPath（按优先级选择）
  - CSS: 速度快、语法简洁
  - XPath: 能定位复杂结构，但性能稍差
- **等待机制**：
  - Implicit Wait — 全局等待，找元素前等 N 秒
  - Explicit Wait — 等待特定条件满足（最推荐）
  - Fluent Wait — 带 polling interval + ignore exceptions
- **Page Object 模式** — 每个页面一个类，封装元素和操作，测试只写流程
- **多浏览器** — WebDriver 接口 + ChromeDriver / FirefoxDriver / EdgeDriver + RemotesDriver (Grid)
- **常用操作**：click / sendKeys / getText / select 下拉框 / Alert / Window 切换 / Frame 切换

**面试重点**：
- Page Object 设计模式几乎是必问
- 等待机制的选择理由（为什么不用 Thread.sleep）
- 如何处理弹窗、新窗口、iframe

### [[rest-assured-api]]

- **核心风格**：Given-When-Then 三段式链式调用

```java
given()
    .header("Authorization", "Bearer xxx")
    .body(requestBody)
.when()
    .post("/api/users")
.then()
    .statusCode(201)
    .body("data.name", equalTo("Alice"));
```

- **JSON Schema 校验**：用 `matchesJsonSchemaInClasspath("schema.json")` 校验响应结构
- **认证处理**：Basic Auth / OAuth2.0 / Token 动态获取
- **参数传递**：提取上一个接口的响应值传给下一个接口（Extract + 变量保存）

### [[junit-testng]]

| 特性 | JUnit 5 | TestNG |
|------|---------|--------|
| 核心注解 | @Test, @BeforeEach, @AfterEach | @Test, @BeforeMethod, @AfterMethod |
| 参数化 | @ParameterizedTest + @CsvSource | @DataProvider |
| 分组 | @Tag | @Test(groups = "...") |
| 依赖测试 | @Order | dependsOnMethods / dependsOnGroups |
| 监听器 | Extensions API | ITestListener / IAnnotationTransformer |

**面试重点**：
- @BeforeClass / @BeforeMethod 的执行顺序
- 数据驱动测试（JUnit5 @CsvSource / @MethodSource vs TestNG @DataProvider）
- 断言库：AssertJ / Hamcrest 的流式断言优于原生 Assert

### [[auto-framework-design]]

企业级自动化框架的核心组成：

1. **驱动管理 (Driver Management)** — WebDriver 统一初始化、复用、销毁，支持多浏览器 + Remote
2. **配置化 (Configuration)** — 环境/浏览器/超时等参数统一在 config 文件中管理 (YAML/Properties)
3. **数据驱动 (Data-Driven)** — Excel / YAML / JSON / CSV 读取测试数据，与用例逻辑分离
4. **重试机制 (Retry)** — 用例失败自动重试 N 次（TestNG IRetryAnalyzer / JUnit5 自定义 Extension）
5. **报告集成 (Reporting)** — Allure / ExtentReports / ReportNG 定制测试报告
6. **日志系统 (Logging)** — Log4j / SLF4J 记录执行过程
7. **持续集成 (CI/CD)** — Jenkins / GitHub Actions / GitLab CI 集成自动化执行
8. **失败截图** — 用例失败时自动截图并附加到报告

**面试重点**：
- "你搭建过框架吗？讲一下框架的核心组成部分" — 这是自动化岗位的高频面试题
- 能说出"为什么选择数据驱动"、"如何处理用例依赖"、"如何在 CI 中集成"

## 四、[[api-testing-basics]]

### HTTP 协议基础

- **请求方法**：GET（查）/ POST（增）/ PUT（全量更新）/ PATCH（部分更新）/ DELETE（删）
- **状态码分类**：
  - 2xx：成功（200 OK、201 Created、204 No Content）
  - 3xx：重定向（301 永久、302 临时、304 未修改）
  - 4xx：客户端错误（400 参数错、401 未认证、403 未授权、404 不存在、405 方法不允许）
  - 5xx：服务端错误（500 内部错误、502 网关超时、503 服务不可用）
- **RESTful 规范**：资源 URL 设计、无状态、使用标准 HTTP 方法
- **鉴权方式**：Basic Auth / Bearer Token / JWT / OAuth2.0 / API Key

### 接口测试关注点

| 关注维度 | 具体检查项 |
|---------|-----------|
| 功能正确性 | 返回状态码、响应体字段值、数据操作正确 |
| 幂等性 | GET / DELETE / PUT 多次请求结果一致 |
| 异常处理 | 参数缺失/类型错误/越界、未认证访问、不存在的资源 |
| 性能 | 接口响应时间是否在预期内（P99 < 500ms）|
| 安全 | SQL 注入／越权（水平越权、垂直越权）|

**面试重点**：
- POST 和 PUT 的区别
- 接口测试如何做参数校验（等价类 + 边界值）
- 如何测试幂等性

## 五、[[performance-testing]]

### 核心术语

| 术语 | 含义 | 说明 |
|------|------|------|
| **QPS** | Queries Per Second | 每秒查询数，常用于读接口 |
| **TPS** | Transactions Per Second | 每秒事务数，常用于写接口 |
| **RT** | Response Time | 响应时间（含网络+服务端） |
| **P99** | 99% 的请求响应时间 | 衡量长尾延迟，P99 < 1s 是常见目标 |
| **并发数** | 同时处理的请求数 | 系统能支撑多少在线用户 |
| **吞吐量** | 系统单位时间处理的请求数 | TPS/QPS |

### 性能测试流程

```
基准测试 → 负载测试 → 压力测试 → 稳定性测试
   ↓          ↓           ↓           ↓
单用户基线  逐步加压   极限施压   长时间运行
```

1. **基准测试 (Baseline)** — 单用户场景，确定性能基线
2. **负载测试 (Load)** — 模拟预期用户数，验证是否达标
3. **压力测试 (Stress)** — 不断加压直到系统崩溃，找到瓶颈
4. **稳定性测试 (Soak)** — 长期运行（如 24h）检测内存泄漏

### JMeter 核心组件

- **线程组 (Thread Group)** — 模拟用户数、Ramp-Up 时间、循环次数
- **Sampler** — HTTP Request / JDBC Request / Debug Sampler
- **断言 (Assertion)** — Response Assertion / Duration Assertion / JSON Assertion
- **监听器 (Listener)** — 查看结果树 / 聚合报告 / 图形结果
- **前置/后置处理器** — 参数提取（JSON Extractor / Regular Expression Extractor）
- **配置元件** — CSV Data Set Config / HTTP Cookie Manager / HTTP Header Manager

### 性能瓶颈定位

```
用户端 → 网络 → 负载均衡 → 应用服务器 → 中间件 → 数据库
                                                    ↓
                                               慢查询/连接池
```

| 层次 | 常见瓶颈 | 排查工具 |
|------|---------|---------|
| 前端 | 资源过大、请求过多 | Chrome DevTools / Lighthouse |
| 网络 | 带宽、DNS 解析 | Wireshark / ping / traceroute |
| 应用 | 慢代码、线程阻塞 | JProfiler / Arthas / YourKit |
| 数据库 | 慢 SQL、索引缺失、连接池满 | MySQL slow log / EXPLAIN / DBA |
| 缓存 | 缓存穿透、击穿、雪崩 | Redis-cli / Grafana |
| 中间件 | 队列积压 | Kafka lag / RabbitMQ 管理台 |

**面试重点**：
- QPS / TPS / RT / P99 的定义和区别
- 能说出"性能测试的完整流程"
- "项目中有没有遇到过性能问题？你们怎么定位的？"

## 六、面试核心攻略

### 高频场景题

**场景 1：登录功能怎么测？**

```
功能类：
  - 正确账号密码能否登录
  - 错误密码提示是否正确
  - 空用户名/密码的校验
  - 密码显示/隐藏切换
  - 记住密码 / 忘记密码流程

安全类：
  - SQL 注入（1' OR '1'='1）
  - XSS（<script>alert(1)</script>）
  - 暴力破解是否有验证码/限制
  - Token 过期是否跳回登录页
  - 登录态下修改密码后旧 Token 是否失效

兼容类：
  - 不同浏览器显示是否正常
  - 手机端响应式适配
  - 输入法兼容（中英文输入）

性能类：
  - 多人同时登录
  - Token 生成时间
```

**场景 2：购物车怎么测？**

```
基础功能：
  - 添加商品/删除商品/修改数量
  - 商品总价计算是否正确
  - 优惠券/满减叠加计算
  - 库存不足时提示

边界/异常：
  - 商品数量为 0、负数、超过库存
  - 商品下架后购物车状态
  - 未登录加入购物车 → 登录后是否还在
  - 多端同步（PC 加了 → App 能看到）

并发：
  - 多人抢购同一商品（超卖问题）
  - 购物车操作和下单同时进行
```

**场景 3：支付流程怎么测？**

```
正向流程：
  - 余额足够时支付成功
  - 余额不足时提示失败
  - 支付成功后订单状态改为已支付
  - 退款流程

异常流程：
  - 支付中网络中断 → 订单状态（等待支付？超时取消？）
  - 重复支付（点了两次支付按钮）
  - 支付超时处理
  - 第三方支付回调失败

安全：
  - 订单金额篡改（拦截请求改金额）
  - 支付结果伪造
  - 重放攻击
```

### STAR 原则回答模板

面试中回答"请讲一个你最印象深刻的 Bug"或"介绍一下你的项目"时，用 **STAR 原则**组织回答：

| 要素 | 含义 | 话术示例 |
|------|------|---------|
| **S**ituation | 项目/场景背景 | "我在 XX 项目中负责订单模块的测试" |
| **T**ask | 你的任务 | "需要覆盖支付全流程，包括退款和异常场景" |
| **A**ction | 你做了什么 | "我用等价类+边界值设计了 50 个用例，发现了支付回调幂等性问题" |
| **R**esult | 结果/成效 | "提前发现了线上生产 Bug，避免了预计影响 10 万用户的故障" |

### 如何展示自动化项目经验

面试时展示自动化经验的核心策略：

1. **先讲项目背景** — 什么项目、团队规模、测试范围
2. **框架设计思路** — 为什么选择这个框架、解决了什么问题
3. **你的具体贡献** — 写了多少用例、覆盖多少接口、发现了什么 Bug
4. **技术细节** — 遇到什么技术难点、怎么解决的（如："框架不支持参数化，我引入了 DataProvider 改造了数据驱动层"）
5. **效果数据** — 自动化覆盖率、执行时间缩短了多少、线上漏测率变化

**"我不会直接说我会 Selenium，而是说：'我基于 Page Object 模式设计了一套自动化框架，覆盖了 300+ 用例，每次回归从 2 天缩短到 2 小时'"**

## 跨知识体系连接

### 与 [[AI应用开发知识体系]] 的连接

| 连接点 | 说明 |
|-------|------|
| **AI 辅助测试用例生成** | 用 LLM（Claude/GPT）基于需求文档自动生成测试用例，测试人员做 Review 和补充 |
| **AI 智能 Bug 分析** | 将日志/错误截图喂给 AI，自动识别错误原因、给出修复建议、分配合适的开发 |
| **视觉回归测试自动化** | 用 AI 做截图对比，解决传统像素对比误报率高的问题（如内容变化但功能正常） |
| **测试数据生成** | 用 AI 生成大量符合业务逻辑的测试数据，替代手工造数据的低效方式 |
| **测试报告智能总结** | 用 AI 将测试结果自动生成自然语言报告，减少人工汇总 |
| **知识库 RAG** | 将历史 Bug 和经验文档做成 RAG 知识库，新测试人员可以快速查询类似问题的处理方式 |

AI 应用开发能力不是替代测试，而是 **让测试人员从重复劳动中解放出来，聚焦在更有价值的探索性测试和测试设计上**。

### 与 [[Java开发知识体系]] 的连接

| 连接点 | 说明 |
|-------|------|
| **Java 是自动化框架的基础语言** | Selenium、Rest Assured、Appium、JMeter 脚本、TestNG/JUnit 全基于 Java |
| **OOP 设计模式** | Page Object、Factory、Builder、Singleton 在框架中广泛应用 |
| **集合与 Lambda** | 用例数据管理、列表筛选、Map 转换、Stream 操作无处不在 |
| **多线程** | 并发执行测试用例、多浏览器并行、性能测试脚本 |
| **Spring Boot** | 自动化框架集成 Spring Boot 做依赖注入和配置管理 |
| **Maven/Gradle** | 测试框架依赖管理、生命周期绑定（test → surefire → report） |
| **SQL** | 接口测试数据准备/清理、数据库断言、性能慢查询定位 |
| **Git** | 版本管理测试代码、CI 触发自动执行、多人协作 |

**不是你一边学测试一边丢了 Java，而是你用 Java 写出更好的测试框架。**

## 学习路径建议

### 第 1 阶段：测试理论基础（2 周）

- 写测试报告

- JUnit/TestNG 参数化

- 从零搭建 Maven + Selenium + TestNG 项目

- 集成 CI（GitHub Actions）
- 用 Docker + Selenium Grid 做分布式执行

### 第 5 阶段：进阶（持续）
- 性能测试（JMeter + 瓶颈分析）
- 持续集成与持续交付
- AI 辅助测试实践
